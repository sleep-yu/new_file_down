一、背景与目标

大文件上传在弱网或长时上传场景下容易失败，整文件一次性重传成本高，且用户无法感知当前进度与恢复状态。为了解决这个问题，需要将上传流程拆分成多个可恢复、可补传、可重试的小单元。

目标能力

支持大文件上传、断点续传、失败重试、上传进度展示、最终文件合并。

运行边界

浏览器上传客户端 + 单机 Upload API Server + 本地磁盘临时存储。

交付结果

冻结前后端协议、状态模型与目录结构，减少后续开发重复决策。

二、设计原则

流程闭环优先 接口幂等 状态可恢复 边界安全校验

  • 先保证主链路闭环：切片、查状态、补传缺片、合并完成。
  • 分片上传必须具备幂等语义，重复提交不会污染状态。
  • 上传状态必须落盘，页面刷新后仍可恢复。
  • 前端负责切片、调度、重试与进度；后端负责落盘、状态记录与合并。
  • 系统边界处执行必要校验：文件名、分片索引、文件大小、请求头、路径安全。

三、整体架构图

上传链路以 fileHash 作为上传会话标识。客户端先计算文件 hash，再查询服务端已完成分片，最后仅补传缺失分片并触发合并。

四、主上传流程

主链路由“算 hash → 查状态 → 补传缺片 → 请求合并”组成，上传状态的唯一真相由服务端状态文件维护。

sequenceDiagram participant U as 用户 participant C as Upload Client participant A as Upload API participant S as temp_chunks/ U->>C: 选择文件 C->>C: 按 5MB 切片\n计算 MD5(fileHash) C->>A: GET /upload-status?fileHash A->>S: 读取 status.json S-->>A: uploadedChunks / isComplete A-->>C: 上传状态 alt 文件已经完成 C-->>U: 直接显示 success else 存在缺失分片 loop 仅上传缺失分片 C->>A: POST /upload-chunk A->>S: 写入 chunk + 更新 status.json A-->>C: success end C->>A: POST /merge-chunks A->>S: 按索引顺序合并分片 A-->>C: isComplete = true C-->>U: 显示上传完成 end

五、断点续传流程

续传依赖服务端状态文件，而不是浏览器本地缓存。用户重新选择同一文件后，客户端重新计算相同的 fileHash 即可恢复上传。

sequenceDiagram participant C as Upload Client participant A as Upload API participant S as temp_chunks/ Note over C: 网络中断 / 页面刷新 / 浏览器重开 C->>C: 重新选择同一文件\n重新计算 fileHash C->>A: GET /upload-status?fileHash A->>S: 读取 uploadedChunks S-->>A: 当前上传状态 A-->>C: uploadedChunks C->>C: diff 缺失分片 loop 仅补传缺失分片 C->>A: POST /upload-chunk A-->>C: success end C->>A: POST /merge-chunks A-->>C: merged success

六、失败重试与恢复流程

首版只对单片上传做有限次自动重试；超过上限则进入错误态，等待用户再次触发上传，并重新查询服务端状态后继续补传。

七、前端状态机图

前端只维护调度状态，不保存服务端权威状态副本；所有恢复都通过重新查询上传状态完成。

八、接口协议

首版协议固定为 3 个接口：查询状态、上传分片、触发合并。所有接口返回 JSON，错误统一使用结构化响应。

8.1 接口总览

接口方法用途关键幂等语义
/upload-statusGET查询已上传分片与完成状态同一 fileHash 多次查询返回同一状态快照
/upload-chunkPOST上传单个分片重复上传同一分片直接返回成功
/merge-chunksPOST按顺序合并所有分片重复合并返回已完成结果

8.2 错误响应结构

{
  "error": {
    "code": "INVALID_REQUEST",
    "message": "human readable message"
  }
}

8.3 GET /upload-status

用途查询 fileHash 对应的已上传分片列表，用于断点续传。
QueryfileHash 必填。
状态码200 成功；400 缺少 fileHash。
{
  "fileHash": "f5d1278e8109edd94e1e4197e04873b9",
  "fileName": "video.mp4",
  "fileSize": 734003200,
  "chunkSize": 5242880,
  "totalChunks": 140,
  "uploadedChunks": [0, 1, 2, 3],
  "isComplete": false,
  "mergedFileName": null,
  "updatedAt": "2026-04-19T12:00:00.000Z"
}

8.4 POST /upload-chunk

HeadersX-File-Hash、X-File-Name、X-Chunk-Index、X-Total-Chunks、X-File-Size、X-Chunk-Size、Content-Type: application/octet-stream
Body当前分片原始二进制内容。
状态码200 成功或幂等命中；400 参数错误；409 元数据冲突；500 写盘失败。
{
  "fileHash": "f5d1278e8109edd94e1e4197e04873b9",
  "chunkIndex": 18,
  "stored": true,
  "alreadyExists": false,
  "uploadedChunksCount": 19,
  "totalChunks": 140,
  "isComplete": false
}

8.5 POST /merge-chunks

BodyfileHash、fileName、fileSize、chunkSize、totalChunks
状态码200 合并成功；400 请求错误；409 分片未齐或元数据冲突；500 合并失败。
{
  "fileHash": "f5d1278e8109edd94e1e4197e04873b9",
  "isComplete": true,
  "mergedFileName": "video__f5d1278e.mp4",
  "uploadedChunksCount": 140,
  "totalChunks": 140
}

九、存储设计

服务端以 fileHash 作为工作目录键，状态文件与分片文件聚合在同一目录；最终合并文件写入 uploads/。

9.3 存储规则

  • 分片文件名固定为 <chunkIndex>.part。
  • 首次接收某个 fileHash 时创建工作目录与初始 status.json。
  • 每次写入新分片后，更新 uploadedChunks 与 updatedAt。
  • 合并时按索引升序流式读取分片，写入临时合并文件，再原子改名到 uploads/。
  • 合并成功后删除 chunks/，保留 status.json 以供短期查询。

十、前后端职责拆分

前端职责

  • 按固定 chunk size 切片。
  • 计算 MD5，得到 fileHash。
  • 调用状态接口，拿到 uploadedChunks。
  • 构造 missingChunks 并执行并发上传。
  • 聚合上传进度，并处理失败重试。
  • 缺片全部上传后触发 merge-chunks。

后端职责

  • 校验请求头、文件名、索引范围和文件元信息。
  • 按 fileHash / chunkIndex 落盘分片。
  • 维护 status.json 与 uploadedChunks。
  • 处理幂等上传与元数据冲突。
  • 按顺序合并最终文件并返回完成状态。
  • 清理临时分片目录与合并锁文件。

十一、关键设计决策

问题决策原因
文件唯一标识使用文件内容 MD5 作为 fileHash同一文件可稳定恢复到同一上传会话
切片大小初版固定为 5MB实现简单，兼顾请求数与恢复粒度
分片索引统一使用 0-based前后端实现一致，数组映射自然
上传协议POST /upload-chunk 走原始二进制 body避免 multipart 包裹，服务端直接按文件写盘
并发数前端固定 4 路并发吞吐与服务端压力之间平衡，适合首版验证
重试策略每片最多重试 3 次，退避 500/1000/2000ms控制失败重试成本，避免无穷重试
合并触发分片全部上传完后显式调用 POST /merge-chunks让合并成为明确步骤，便于恢复与追踪
临时目录统一使用 temp_chunks/与最终文件目录职责分离，便于清理
重复上传分片返回成功，不重复记账保证网络抖动场景的幂等语义

十二、非目标

  • 不覆盖鉴权、权限隔离、多租户权限控制。
  • 不覆盖对象存储、CDN、多节点共享磁盘或分布式协调。
  • 不引入数据库保存上传状态。
  • 不设计暂停/继续、多文件批量上传、秒传判重等扩展能力。

十三、验收标准

  • 文档为单页 HTML，可直接浏览和评审。
